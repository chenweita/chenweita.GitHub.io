# 使用意义
1. 提升代码质量与可靠性‌：通过编写测试用例，Jest 帮助开发者验证代码是否按预期工作，及时发现潜在错误，从而减少生产环境中的 bug，提高整体代码质量。‌

2. ‌简化测试流程‌：Jest 提供了开箱即用的简单 API 和配置，支持自动发现和运行测试，无需复杂设置即可快速上手，降低了测试的入门门槛。‌

3. ‌强大的功能集成‌：Jest 内置了多种实用功能，如断言库、模拟（Mock）函数、快照测试、代码覆盖率报告等，能够满足单元测试、集成测试等多种场景需求，提升测试效率。‌

4. ‌高效执行与并行处理‌：Jest 支持并发测试运行，对模块进行隔离和缓存，避免重复执行，显著加快测试速度，尤其适合大型项目。‌

5. ‌促进开发实践优化‌：通过与持续集成（CI/CD）流程结合，Jest 能在代码提交后自动运行测试，帮助团队及时发现问题，推动测试驱动开发（TDD）和高质量代码规范的落实。‌

6. ‌良好的社区与生态支持‌：作为开源项目，Jest 拥有活跃的社区和丰富的插件生态系统，便于扩展和维护，确保长期可用性。‌

# 核心概念与基本用法
Jest 测试文件通常以 .spec.js 或 .test.js 为后缀，使用 test() 或 it() 函数定义测试块。每个测试块包含一个描述字符串和一个回调函数，其中使用 expect() 进行断言，配合匹配器如 toBe() 验证预期结果。‌

断言与匹配器‌：expect() 包装被测函数的输出，匹配器如 toBe()、toEqual() 等用于比较实际值与预期值。如果不符合预期，测试会失败并抛出错误。‌


test('1+1=?', () => {
    expect(sum(1,1)).toBe(3)
})


模拟（Mocking）‌：Jest 的 jest.mock() 方法可以模拟模块或函数的实现，常用于隔离依赖。例如，模拟文件读取操作：
jest.mock('fs', () => ({
  readFile: jest.fn((path, callback) => callback(null, 'mocked data')),
}));


# 支持异步
function addAsync(a, b) => {
    return new Promise((reslove) => {
        setTimeout(() => {
            return a + b
        }, 1000)
    })
}
test('1 + 1 async', () => {
    addAsync(1, 1).then(res => {
        expect(res).toBe(2)
    }) 
})

# Mock与Spy
Jest中的Mock和Spy是用于模拟和替换依赖的一个功能。Mock可以模拟一个依赖函数，Spy可以用于监视和断言一个函数的行为。

## mock
jest.mock第一个参数为文件路径，第二个参数是一个对象，key为函数名，value为jest.fn().mockReturnValue(xxx)
    jest.mock('./math', () => ({
        add: jest.fn().mockReturnValue(111)
    }))


## Spy示例：

        // src/math.js
        function add(a, b) {
        return a + b;
        }

        // tests/example.test.js
        const { add } = require('../src/math');
        const addSpy = jest.spyOn(require('../src/math'), 'add');

        test('add should use spy', () => {
        const result = add(1, 2);
        expect(addSpy).toHaveBeenCalled();
        expect(result).toBe(3);
        });

# 运行测试与高级功能
1. 命令行操作‌：通过 npm run test 或 npx jest 运行测试，支持参数如 --coverage 生成覆盖率报告。‌
2. ‌快照测试‌：使用 snapshotSerializers 生成组件或数据的快照，后续测试会对比快照是否变化。‌
3. ‌测试环境‌：Jest 默认使用 jsdom 模拟浏览器环境，可通过 testEnvironment 配置调整。‌
4. ‌覆盖率报告‌：启用 collectCoverage 后，Jest 会在 coverageDirectory 输出 HTML 或 JSON 格式的覆盖率文件，帮助识别未测试代码。‌


# 并行测试
Jest支持并行执行测试用例，能够显著减少测试用例的运行时间。

在package.json中添加Jest配置：

{
  "jest": {
    "maxWorkers": "50%"
  }
}


# Jest配置与最佳实践
配置文件详解
Jest使用jest.config.js或package.json中的jest键值对进行配置。常用的配置选项包括collectCoverage、coverageDirectory、coverageReporters等。

示例：

{
  "jest": {
    "collectCoverage": true,
    "coverageDirectory": "<rootDir>/coverage",
    "coverageReporters": ["html", "text"]
  }
}
# 自定义配置
根据项目需求，可以对Jest进行自定义配置。例如，可以自定义测试文件的匹配模式、忽略的文件、测试环境等。

示例：

{
  "jest": {
    "testMatch": ["<rootDir>/src/**/*.{js,jsx,ts,tsx}"],
    "testIgnore": ["<rootDir>/src/**/*.test.js"],
    "testEnvironment": "node"
  }
}
测试代码组织与维护
良好的测试代码组织和维护是保持项目稳定性的关键。建议遵循以下最佳实践：

按模块组织文件：将测试文件组织在与源代码一致的目录结构中。例如：

/src
├── math.js
└── anotherModule.js
/tests
├── math.test.js
└── anotherModule.test.js
命名规范：使用.test.js或.spec.js作为测试文件的后缀。
保持简洁：每个测试文件尽量只测试一个功能模块。
定期清理：定期清理不再需要的测试文件，保持测试代码的整洁和有效。
